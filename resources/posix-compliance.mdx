BashKit provides substantial compliance with IEEE Std 1003.1-2024 (POSIX.1-2024) Shell Command Language specification.

## Design Philosophy

BashKit prioritizes:

1. **Security over completeness** - Exclude features that break sandbox containment
2. **Stateless execution** - No persistent state between command invocations
3. **Deterministic behavior** - Predictable results for AI agent workflows

## Security Exclusions

Two POSIX special builtins are intentionally excluded for security reasons:

### exec

The POSIX `exec` replaces the current shell process, which would break sandbox containment. Scripts requiring `exec` should be refactored to use standard command execution.

**Rationale:** Replacing the shell process would escape the virtual environment and allow arbitrary host process execution.

**Alternative:** Use direct command execution or script invocation within the sandbox.

### trap (Partial)

Signal handlers require persistent state across commands, conflicting with Bashkit's stateless execution model. Additionally, there are no signal sources in the virtual environment (no external processes send SIGINT/SIGTERM).

**Rationale:** No signal delivery mechanism exists in the virtual environment.

**Status:** EXIT and ERR traps are implemented. Signal traps (INT, TERM, etc.) are stored but never triggered.

**Alternative:** Handle errors through exit codes and conditional execution.

## Intentional Deviations

### Security-Motivated

1. **No OS process spawning** - External commands run as builtins or virtual script re-invocations
2. **No signal handling** - `trap` signals excluded for sandbox isolation
3. **No process replacement** - `exec` excluded for containment
4. **Virtual filesystem** - Real FS access requires explicit configuration
5. **Network allowlist** - HTTP requires URL allowlist configuration

### Simplification (Stateless Model)

1. **Background execution** - `&` is parsed but runs synchronously
2. **Job control** - Not implemented (interactive feature)
3. **Process times** - `times` returns zeros (no CPU tracking)

## Word Expansions

All POSIX word expansions are supported:

- Tilde expansion (`~`)
- Parameter expansion (`$var`, `${var}`)
- Command substitution (`$(cmd)`)
- Arithmetic expansion (`$((expr))`)
- Field splitting
- Pathname expansion (globbing)
- Quote removal

## Redirections

All POSIX redirection operators are supported:

- `<` - Input redirection
- `>` - Output redirection
- `>>` - Append output
- `<<` - Here-document
- `<<-` - Here-document with tab stripping
- `<<<` - Here-string (Bash extension)
- `2>` - Stderr redirection
- `2>&1` - Stderr to stdout
- `&>` - Both to file (Bash extension)

## Functions

Both POSIX and Bash-style function definitions are supported:

```bash
# POSIX style
name() {
    command
}

# Bash style
function name {
    command
}
```

## Testing Approach

POSIX compliance is verified through:

1. **Unit Tests** - 22 POSIX-specific tests for special builtins and parameters
2. **Spec Tests** - 435+ bash spec test cases in CI
3. **Bash Comparison** - Differential testing against real bash

### Running Tests

```bash
# Run POSIX compliance tests
cargo test --lib -- interpreter::tests::test_colon
cargo test --lib -- interpreter::tests::test_readonly
cargo test --lib -- interpreter::tests::test_eval

# Compare with real bash
cargo test --test spec_tests -- bash_comparison_tests --ignored
```

|-----------|-------------|

| `$*` | Positional parameters (joined) |
| `$#` | Number of positional parameters |
| `$?` | Last exit status |
| `$-` | Current option flags |
| `$$` | Process ID |
| `$!` | Last background job ID |
| `$0` | Shell or script name |
